
# 🏦 Integrated Account Management System (통합 계좌 관리 시스템)
Python과 Oracle DB를 활용하여 구축한 사용자 중심의 안전한 금융 서비스 시스템입니다. 14가지 핵심 금융 요구사항을 분석하고 이를 데이터베이스 설계 및 비즈니스 로직에 완벽히 반영하여 구현하였습니다.

# 📑 목차
1. 프로젝트 개요
2. 주요 기능 (14가지 요구사항)
3. 데이터베이스 설계 (ERD)
4. 기술적 핵심 (트랜잭션 및 보안)
5. 시스템 시연 시나리오

# 🚀 프로젝트 개요
● 개발 환경: Python

● 데이터베이스: Oracle DB 

● 설계 도구: Draw.io (ERD 설계) 

● 주요 특징: 사용자(USERS) 중심의 1:N 관계 설계를 통한 개인화된 계좌 관리 및 실시간 거래 추적

# ✨ 주요 기능 (14가지 요구사항)
보안, 기능성, 안정성을 테마로 한 14가지 금융 서비스 요구사항을 준수합니다. 

# 🔐 사용자 인증 및 권한
● 로그인 기반 서비스: 회원 인증 후 모든 서비스 접근 가능 

● 관리자 전용 메뉴: 관리자(is_admin) 계정에만 사용자 정보 조회·수정·삭제 권한 부여 

# 🏦 계좌 관리 및 제약 조건
● 데이터 표준화: 은행명, 계좌번호, 잔액, 별칭 중심의 구조화 

● 은행 등록 제한: 하나, 우리, 국민, 신한, 기업은행 등 5대 주요 은행만 등록 가능 

● 생성 제약: 계좌 개설 시 최소 1,000원 이상의 잔액 필요 유효성 검사: 별칭 수정 시 중복 방지 로직 적용 

# 💸 금융 거래 및 안정성
● 거래 서비스: 입금, 출금, 계좌이체 기능 제공 

● 이체 유효성: 0원 초과 금액만 거래 가능하며, 잔액 부족 시 자동 차단 

● 데이터 무결성: 모든 거래(입출금/이체) 내역을 실시간으로 기록 및 추적

# 📊 데이터베이스 설계 (ERD)
사용자-계좌-거래 내역으로 이어지는 3단계 연결 구조를 통해 데이터 무결성을 보장합니다. 

● Users (사용자): 아이디(PK), 비밀번호, 이름, 연락처, 관리자 여부 

● Account (계좌): 계좌번호(PK), 사용자ID(FK), 은행명, 잔액, 별칭, 생성일 

● Transaction (계좌 내역): 내역번호(PK), 계좌번호(FK), 거래유형, 거래금액, 상대계좌, 거래일시

# 🛡️ 트랜잭션 원자성 보장 (Atomic Transaction)
계좌이체와 같이 여러 단계의 작업이 필요한 경우, 전체 성공 시에만 Commit하고 하나라도 실패할 경우 전체 Rollback을 수행하여 데이터 일관성을 유지합니다.

# 🎬 시스템 시연 시나리오
시스템의 안정성을 검증하기 위해 총 6단계의 시나리오를 수행합니다. 
1. 사용자 접속: 로그인 및 세션 유지, 관리자 권한별 메뉴 분기 
2. 계좌 생성 및 예외 처리: 5대 은행 외 등록 제한 및 최소 금액 검증 
3. 조회 및 수정: 별칭 수정 및 실시간 잔액 반영 확인 
4. 입출금: 실시간 잔액 변동 및 내역 자동 기록 
5. 계좌이체 및 무결성: 잔액 초과 이체 차단 및 3단계 트랜잭션 검증 
6. 관리자 기능: 관리자 전용 메뉴를 통한 전체 사용자 정보 관리
